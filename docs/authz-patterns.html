CouchDB — 3 Authorization Patterns

เจาะลึกแต่ละ pattern สำหรับ scope user เข้าถึงเฉพาะบาง database / document พร้อม code จริงและ trade-offs

บทนำ

CouchDB มี authorization 2 ระดับ native: database-level (ผ่าน _security doc) และ document-level (ผ่าน validate_doc_update function) ถ้าต้องการละเอียดกว่านั้น ต้องวาง backend proxy คั่น สามแนวทางนี้ไม่ได้แทนกันเสมอ — มักผสมกันตามความต้องการจริง

คำถามหลักที่ต้องตอบก่อนเลือก: granularity ที่ต้องการคืออะไร? (ทั้ง database / บาง document / บาง field) และ client sync ตรงกับ CouchDB หรือผ่าน backend? ถ้า client sync ตรง (PouchDB ↔ CouchDB) → ถูกจำกัดที่ Pattern 1 + 2 เท่านั้น
← กลับไปเอกสารสถาปัตยกรรมหลัก

Pattern 1 · Database-per-scope + Roles

enforce: CouchDB native granularity: database setup: medium

แต่ละ "หน่วยของสิทธิ์" = หนึ่ง database แยกกัน คุมการเข้าถึงด้วย roles ใน _security doc ของ DB นั้น — ไม่ใช้ names เพราะ names ต้องแก้ทุกครั้งที่เพิ่มคน ส่วน roles แก้ที่ user document ที่เดียว

ทำงานยังไง

CouchDB ตรวจตอนทุก request: เอา userCtx.roles มา intersect กับ members.roles ของ DB ถ้ามีตัวร่วมอย่างน้อย 1 → ผ่าน ถ้าไม่มี → 403 (_admin ผ่านเสมอ)

รูปที่ 2 — Roles intersection check
sequenceDiagram participant C as Client (PouchDB) participant DB as CouchDB C->>DB: GET /branch1_stock (cookie) DB->>DB: userCtx.roles = ["branch1:stock"] DB->>DB: members.roles = ["branch1:stock"] DB->>DB: intersect ≠ ∅ ? alt มีตัวร่วม DB-->>C: 200 + docs else ไม่มี DB-->>C: 403 Forbidden end

Code

1. ตั้ง role ที่ user document

// PUT /_users/org.couchdb.user:nai_a
{
  "name": "nai_a",
  "password": "...",
  "roles": ["branch1:stock"],
  "type": "user"
}

2. ตั้ง _security ที่แต่ละ database

// PUT /branch1_stock/_security
{
  "admins":  { "names": [], "roles": ["branch1:manager"] },
  "members": { "names": [], "roles": ["branch1:stock"] }
}

// PUT /branch1_bill/_security
{
  "admins":  { "names": [], "roles": ["branch1:manager"] },
  "members": { "names": [], "roles": ["branch1:bill"] }
}

3. Client sync ตรงได้เลย — ไม่ต้องเขียน auth logic

// นาย A เปิดได้เฉพาะ DB ที่ role ครอบคลุม
const stock = new PouchDB('http://host:5984/branch1_stock', {
  fetch: (url, opts) => fetch(url, { ...opts, credentials: 'include' })
});
stock.sync(localStock, { live: true, retry: true });
// branch1_bill → 403 ตั้งแต่ handshake

ข้อดี / ข้อเสีย

✅ ข้อดี

  • CouchDB enforce เอง ไม่ต้องเขียน code security
  • read + write ถูกกรองอัตโนมัติพร้อมกัน
  • per-database replication → sync เฉพาะ scope
  • DB พังตัวนึงไม่ลามตัวอื่น

❌ ข้อเสีย

  • DB เยอะถ้า branch × resource เยอะ (provision/monitor)
  • query ข้าม DB ทำไม่ได้ตรงๆ ต้องรวมที่ central
  • เพิ่ม branch ใหม่ = ต้อง provision DB ชุดใหม่
เหมาะกับ: ร้านอาหารหลายสาขา, multi-tenant ที่ tenant แยกข้อมูลชัด, shelter management (1 DB ต่อ subsystem ต่อศูนย์) — กรณีที่ขอบเขตข้อมูลตรงกับขอบเขตสิทธิ์

Pattern 2 · validate_doc_update (Document-level)

enforce: CouchDB native granularity: document setup: complex

DB เดียวเก็บข้อมูลหลาย scope ปนกัน ใช้ JavaScript function ใน design document กรอง write ระดับ document — ทุกครั้งที่ write CouchDB เรียก function นี้ ถ้า throw = ปฏิเสธ

ทำงานยังไง

Code

// PUT /stock/_design/access
{
  "_id": "_design/access",
  "validate_doc_update": "function (newDoc, oldDoc, userCtx) {
    if (userCtx.roles.indexOf('_admin') !== -1) return;       // admin ผ่าน
    var branch = newDoc.branch_id;
    if (!branch) throw({ forbidden: 'branch_id required' });
    var required = 'branch' + branch + ':stock';
    if (userCtx.roles.indexOf(required) === -1) {
      throw({ forbidden: 'no access to branch ' + branch });
    }
    // กัน user ย้าย doc ข้าม branch
    if (oldDoc && oldDoc.branch_id !== newDoc.branch_id) {
      throw({ forbidden: 'cannot move across branches' });
    }
  }"
}

ปัญหาสำคัญ — read ไม่ถูกกรอง

validate_doc_update คุมแค่ write การ read (_all_docs, view) ยังคืนทุก document ใน DB ต้องแก้ด้วยอย่างใดอย่างหนึ่ง:

  • Filtered replication — sync เฉพาะ doc ที่ตรง scope (กรองตอน sync ไม่ใช่ตอน enforce)
  • ไม่กันคนตั้งใจ peek — filtered replication ป้องกัน data ที่ sync ลงเครื่อง แต่คนที่ query API ตรงยังเห็นได้ → ถ้าต้องการ read security จริงต้องไป Pattern 1 หรือ 3
// _design/app — filter function
{
  "filters": {
    "by_branch": "function (doc, req) {
      return doc.branch_id == req.query.branch_id;
    }"
  }
}

// client
PouchDB.replicate(remote, local, {
  filter: 'app/by_branch',
  query_params: { branch_id: '1' }
});
ระวัง: Pattern 2 ให้ write security จริง แต่ read security เป็นแค่ "ไม่ส่งมาให้" ไม่ใช่ "ห้ามเห็น" — ถ้าข้อมูลอ่อนไหว (เช่น bill ข้ามสาขา) อย่าพึ่ง Pattern 2 อย่างเดียว

ข้อดี / ข้อเสีย

✅ ข้อดี

  • DB เดียว — query/aggregate รวมง่าย
  • write rule ยืดหยุ่น (ตรวจ field, ความสัมพันธ์ doc)
  • ไม่ต้อง provision DB ต่อ scope

❌ ข้อเสีย

  • read ไม่ปลอดภัยโดยตัวมันเอง
  • function เขียน/debug ยาก (รันใน query server)
  • เปลี่ยน rule = ต้อง update design doc + อาจ rebuild
  • conflict ปนกันใน DB เดียวจัดการยากกว่า
เหมาะกับ: data ที่ scope ปนกันโดยธรรมชาติและ read ไม่ต้องลับมาก เช่น activity log, audit trail, shared catalog ที่ทุกคนอ่านได้แต่เขียนเฉพาะของตัว

Pattern 3 · Backend Proxy

enforce: backend code granularity: ไม่จำกัด setup: complex

วาง API service คั่นระหว่าง client กับ CouchDB — client ไม่คุย CouchDB ตรง proxy ตรวจ JWT/session, map เป็น permission, แล้ว forward เฉพาะที่อนุญาต ใช้เมื่อต้องการ logic ที่ CouchDB native ทำไม่ได้ เช่น field-level, row-level บน query, rate-limit, audit รวมศูนย์

ทำงานยังไง

รูปที่ 4 — Proxy คั่นกลาง
sequenceDiagram participant C as Client participant P as Proxy API participant DB as CouchDB C->>P: GET /api/stock (JWT) P->>P: verify JWT P->>P: map user → permissions alt มีสิทธิ์ P->>DB: query (admin creds) DB-->>P: docs P->>P: filter fields ที่ห้ามเห็น P-->>C: 200 + docs ที่กรองแล้ว else ไม่มีสิทธิ์ P-->>C: 403 end Note over C,DB: client ไม่เห็น CouchDB เลย

Code (sketch)

// FastAPI / Express style pseudo
app.get('/api/stock', auth, async (req, res) => {
  const user = req.user;                       // จาก JWT
  if (!can(user, 'stock:read', req.query.branch))
    return res.status(403).end();

  const docs = await couch.query('stock', { branch: req.query.branch });

  // field-level: ซ่อน cost ถ้าไม่ใช่ manager
  const visible = docs.map(d =>
    can(user, 'stock:cost', req.query.branch) ? d : omit(d, ['cost'])
  );
  res.json(visible);
});
Trade-off ใหญ่: proxy คั่นกลาง = เสีย offline-first client sync ตรงกับ CouchDB ไม่ได้แล้ว ถ้าต้องการทั้ง fine-grained auth และ offline ต้องออกแบบ replication path แยก (เช่น proxy เฉพาะ sensitive ops, sync ตรงสำหรับ data ทั่วไป)

ข้อดี / ข้อเสีย

✅ ข้อดี

  • granularity ไม่จำกัด — field, row, computed
  • รวม business logic, audit, rate-limit ที่เดียว
  • ซ่อน CouchDB จาก client ทั้งหมด

❌ ข้อเสีย

  • เสีย offline-first sync ตรง
  • เพิ่ม service ที่ต้อง deploy/scale/ดูแล
  • เป็น single point of failure ถ้าไม่ HA
  • เขียน auth logic เองทั้งหมด = พื้นที่ bug
เหมาะกับ: ระบบที่ต้อง field-level / computed permission, integrate หลาย backend, หรือ compliance ที่ต้อง audit ทุก access — และ ไม่ ต้องการ offline sync ตรง
